# 팀 커밋 스킬

변경사항을 분석하고 팀 커밋 컨벤션에 맞는 메시지로 커밋한다.

## 커밋 메시지 형식

```
타입:[이름] 한국어 설명
```

예시: `feat:[홍길동] 리마인더 CRUD API 구현`

## 커밋 타입

| 타입 | 용도 |
|------|------|
| basic | 기능은 아니나 프로젝트 관련 코드/로직 추가 |
| init | 초기화 |
| feat | 기능 추가 |
| update | 기능 보완(업그레이드) |
| fix | 버그 수정 |
| hotfix | 긴급 버그 수정, 취약점, 성능 이슈 패치 |
| style | 스타일 변경 (코드 형식, 세미콜론 등 - 로직 변경 없음) |
| refactor | 리팩토링 |
| rename | 이름 변경(파일명, 폴더명, 변수명 등) 또는 파일 이동만 수행 |
| docs | 문서 관련 (README, .gitignore 등 추가/수정/삭제) |
| chore | 기타 변경사항 (빌드 스크립트, 코드 컨벤션 정리 등) |
| remove | 파일 삭제 작업만 수행 |
| test | 테스트 코드 추가/수정/삭제 (로직 변경 없음) |

## 실행 순서

### 1. 이름 확인

`.claude/commit-config.md` 파일에서 이름을 읽는다.

```yaml
---
name: 홍길동
---
```

파일이 없거나 name이 비어있으면, 사용자에게 이름을 물어본 후 `.claude/commit-config.md` 파일을 생성하여 저장한다. 다음부터는 자동으로 읽힌다.

### 2. 변경사항 분석

아래 git 명령어를 실행해서 변경사항을 파악한다:

```bash
git status
git diff --staged
git diff
```

변경사항이 없으면 "커밋할 변경사항이 없습니다"라고 알려준다.

### 3. 커밋 타입 결정

사용자가 타입을 직접 지정한 경우 (예: `/commit feat`) 해당 타입을 사용한다.

지정하지 않은 경우, 변경사항을 분석해서 적절한 타입을 자동 선택한다:

- 새 기능 코드 추가 → `feat`
- 기존 기능 개선/보완 → `update`
- 버그 수정 → `fix`
- 테스트 코드만 변경 → `test`
- 문서만 변경 → `docs`
- 파일 삭제만 → `remove`
- 파일/변수 이름 변경만 → `rename`
- 코드 포맷팅만 → `style`
- 구조 개선 (동작 동일) → `refactor`
- 초기 설정/프로젝트 셋업 → `init`
- 빌드/설정 파일 변경 → `chore`

여러 타입에 해당하는 경우 가장 핵심적인 변경에 맞는 타입을 선택한다.

### 4. 커밋 메시지 작성

- 설명은 한국어로 작성
- 간결하게 핵심만 (1줄, 50자 이내 권장)
- "~구현", "~추가", "~수정", "~삭제" 등 명사형 종결

### 5. 커밋 실행

변경된 파일을 staging하고 커밋한다. `.env`, 시크릿 파일 등 민감한 파일은 제외한다.

커밋 메시지 형식:
```bash
git commit -m "타입:[이름] 설명"
```

커밋 완료 후 결과를 간단히 보여준다 (커밋 해시, 변경 파일 수).
